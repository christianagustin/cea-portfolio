Live Seller Mastersheet Populator

An automated data pipeline that joins three separate business systems into a single, always-current mastersheet — replacing hours of daily VLOOKUP work with a scheduled workflow that runs without human input.

The Problem

Sales and operations teams in high-growth companies almost always hit the same wall. The data they need is split across multiple systems: a CRM for account records, a campaign tool for performance data, and a merchant master list. None of these systems talk to each other natively.

The workaround is a mastersheet: a manually assembled spreadsheet where someone runs VLOOKUP functions, copies and pastes columns, and maintains a merged view. This takes 1-3 hours daily. It is always slightly out of date. When the person who maintains it is on leave, it does not get updated at all.

The most dangerous spreadsheet in your business is the one that only one person knows how to maintain.

How It Works

An in-memory hashmap join. All three sources are loaded into memory at the start of each run. Joining happens in a single code node — no per-row API calls, no rate limit issues.

Load Source ACampaign data fetched from the active campaign sheet. This is the primary owner of operational fields: assigned AM, contact email, contact phone, subscription type.
Load Source BCRM records fetched from Salesforce. This source owns package tier and revenue dates. Where a merchant has multiple CRM records, the one with the latest revenue end date wins.
Load Source CActive merchant list fetched as a fallback. If a merchant exists here but not in Sources A or B, their basic identity data is still captured.
Join in MemoryA code node builds a hashmap keyed on user ID from each source. A second pass merges the three maps with Source A taking priority over B, and B over C.
Smart MergeFor merchants with multiple records (e.g. two CRM entries with different emails), the merge picks the best value — non-null email from any source, latest revenue date for commercial fields.
Change DetectionBefore writing, each candidate row is compared to the current mastersheet row. Only rows where at least one field changed are written. Protected columns are never overwritten.
Batch WriteConfirmed updates are written 50 rows at a time to stay within API rate limits.

Protected Columns

Some fields in the mastersheet are business-owned — they should never be overwritten by automation, regardless of what the source systems say.

Protected
Company name, registration number, CRM account ID, manually entered notes
Owned by Source A
AM assignment, contact email, phone, subscription type
Owned by Source B
Package tier, revenue end date, main category
Fallback (Source C)
Username, basic identity where A and B have no match

Outcome & Impact

  • The mastersheet updates automatically on a schedule. No one needs to run VLOOKUPs.
  • Data is current as of the last scheduled run — not as of last Tuesday when someone had time.
  • Single-person dependency eliminated. If the previous maintainer leaves, the automation keeps running.
  • Manual annotations in protected columns are preserved across every automated update.
